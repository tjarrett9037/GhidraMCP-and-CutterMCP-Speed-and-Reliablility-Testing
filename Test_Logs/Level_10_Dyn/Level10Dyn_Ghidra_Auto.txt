Data read (critical):
- In FUN_00101eb2:
  - hardcoded 0x34-byte target array (52 bytes) into stack `acStack_78`
  - copies 52 bytes from user input into `acStack_b8`
  - calls `FUN_001014a8(acStack_b8, 0x34, 0x123456789abcdef0)`
  - compares transformed input bytes against target bytes; mismatch => return 0, full match => return 1
  - timing and anti-analysis checks present (`signal/raise`, `clock_gettime`, other function calls)
- In FUN_001014a8:
  - derives round material from key and DAT_00103020 via FUN_001013f1
  - applies block operation in counter-like fashion over input (8-byte chunks)
  - XORs generated keystream with input bytes
- In FUN_00101322 + FUN_0010128d:
  - 16-round Feistel-like structure consistent with Blowfish-style design
  - `F()` uses a simplified S-box from DAT_001050c0
- In FUN_0010124d:
  - initializes S-box as `S[i] = i * 0x01010101`
- In FUN_001013f1:
  - copies 0x48-byte P-array from DAT_00103020
  - XORs it with repeated key bytes
- In FUN_00101239:
  - sets DAT_00105098 = 0 (affects zeroization branch behavior)

Interpretation and why it mattered:
- The password check is not plain strcmp with constant string.
- It is "transform(input) == fixed_target", so recovering input requires inverting transform.
- Because transform is XOR with generated keystream: input = target XOR keystream.
- Required ingredients:
  1) target 52-byte array from FUN_00101eb2
  2) key 0x123456789abcdef0
  3) P-array bytes at DAT_00103020 from .rodata
  4) S-box rule from FUN_0010124d
