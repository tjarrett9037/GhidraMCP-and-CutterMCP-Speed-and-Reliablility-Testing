Ransomware Password Recovery Log
Date: 2026-04-08
Analyst: Cursor coding agent (with Ghidra + Cutter MCP tools)

Objective
- Find a password in the loaded ransomware sample that appears to shut down / satisfy the malware's password gate.
- Use both Ghidra and Cutter to increase speed while cross-validating accuracy.

Method Summary
- I treated this as a static reversing workflow:
  1) find candidate strings and imports,
  2) locate xrefs into likely argument-check logic,
  3) decompile the validator,
  4) reverse the transformation routine,
  5) recover and validate plaintext password length and exact compared bytes.

Detailed Step-by-Step Execution, Data Read, Interpretation, and Why

Step 1: Initial string triage in Ghidra + imports in both tools
Commands/queries run:
- Ghidra list_strings(filter="pass")
- Ghidra list_strings(filter="shut")
- Ghidra list_strings(filter="kill")
- Ghidra list_strings(filter="stop")
- Cutter list_imports
- Ghidra list_imports

Raw data returned:
- Ghidra string hit (filter="pass"):
  0x00477042: "Usage: %s <password>\n"
- Ghidra string hits (other filters):
  0x0047a578: "tcache_thread_shutdown(): unaligned tcache chunk detected"
  0x0047b4d8: "Cannot send after transport endpoint shutdown"
  0x0047e245: "ESHUTDOWN"
  0x0047b530: "Operation not possible due to RF-kill"
- Cutter imports: no useful output in this binary context (empty listing in tool response).
- Ghidra imports: no useful output in this binary context (empty listing in tool response).

Interpretation:
- "Usage: %s <password>\n" was a high-confidence anchor indicating command-line password checking.
- The shutdown/kill strings appeared to be generic runtime/libc/network strings, not bespoke malware control strings.
- Imports did not provide immediate clarity, so string xrefs became the fastest pivot.

Why this step:
- Fastest way to locate authentication gates is searching for password-like UI strings and usage messages.

Step 2: Cross-reference the usage/password string
Commands/queries run:
- Ghidra get_xrefs_to(address=0x00477042)
- Cutter xrefs_to(address=0x00477042)
- Ghidra search_functions_by_name("main")
- Cutter search_functions_by_name("main")

Raw data returned:
- Ghidra xrefs to 0x00477042:
  From 0x00401b95 in FUN_00401b3b [DATA]
  From 0x00401b9f in FUN_00401b3b [PARAM]
- Cutter xref:
  (nofunc) 0x401b95 [DATA] lea rdx, str.Usage:__s__password
- search for "main": no direct main symbol discovered.

Interpretation:
- Function at 0x00401b3b likely behaves as entry/dispatcher for CLI args.
- Lack of main symbol is common in stripped binaries; address-based analysis is appropriate.

Why this step:
- Xrefs convert string clues into executable control-flow and pinpoint where the password path branches.

Step 3: Decompile and disassemble 0x00401b3b to identify validator call
Commands/queries run:
- Ghidra decompile_function_by_address(0x00401b3b)
- Cutter decompile_function_by_address(0x00401b3b)
- Ghidra disassemble_function(0x00401b3b)

Raw data read (key points):
- Decompiled logic from Ghidra:
  if argc == 2:
    local_c = FUN_00401a09(argv[1]);
    if local_c == 1 -> print "You achieved Level 4!"
    else -> print "Sorry, not quite there yet."
  else:
    printf("Usage: %s <password>\n", argv[0]);
- Disassembly confirms:
  - argc compared to 2
  - argv[1] passed into call at 0x00401a09
  - success/failure strings selected based on return value

Interpretation:
- 0x00401a09 is the password validator.
- Return value semantics: validator returns true/equal for success branch.

Why this step:
- Needed to isolate the one function that directly compares user input.

Step 4: Decompile/disassemble validator function 0x00401a09 in both tools
Commands/queries run:
- Ghidra decompile_function_by_address(0x00401a09)
- Ghidra disassemble_function(0x00401a09)
- Cutter decompile_function_by_address(0x00401a09)
- Cutter disassemble_function(0x00401a09)

Raw data read (core logic):
- Function initializes 49 bytes of embedded data (stack bytes from 0xBF, 0x2B, ..., 0x16).
- Calls:
  1) copy/memmove-like thunk at 0x00401030 with length 0x31
  2) transformation function FUN_00401725(buffer, 0x31, 0x1d6796e070c749b7)
  3) compare thunk at 0x00401070(buffer, user_input, 0x31)
- Returns comparison == 0.

Interpretation:
- Validator stores encrypted/obfuscated expected bytes in-function.
- Applies deterministic stream transform using 8-byte key constant.
- Compares transformed bytes with user input over exactly 0x31 (49) bytes.

Why this step:
- To recover plaintext password, transformation routine needed full reverse understanding.

Step 5: Reverse transformation routine at 0x00401725
Commands/queries run:
- Ghidra decompile_function_by_address(0x00401725)
- Cutter decompile_function_by_address(0x00401725)
- Also decompiled 0x00401030 and 0x00401070 (thunks/ifunc wrappers).

Raw data read (algorithm characteristics):
- Initializes S-box array 0..255
- KSA-like loop using 8-byte key material (mod 8 indexing)
- PRGA-like loop generating keystream for length N (N = 0x31)
- XORs keystream with buffer bytes in place
- This is RC4-like stream cipher behavior.

Interpretation:
- Since operation is XOR with generated keystream, encryption and decryption are symmetric.
- Therefore, applying same RC4-like transform with same key to embedded ciphertext yields plaintext expected input.

Why this step:
- This is the mathematical path to derive the exact password bytes.

Step 6: Reconstruct and execute decoder script in Python
Command run:
- Python script reproducing RC4-like KSA+PRGA exactly as decompiled, with:
  - encrypted byte array of 50 listed bytes from function initialization
  - key = 0x1d6796e070c749b7, interpreted little-endian to match memory layout

Raw output:
- bytes(out) -> b'flag_{Lin4_D3bugg1ng_iS_MucH_f@5t3r_For_th1$_On3!}'
- utf-8 decode -> flag_{Lin4_D3bugg1ng_iS_MucH_f@5t3r_For_th1$_On3!}

Interpretation:
- This yields a brace-closed string with total length 50.
- However, validator compare length is 49 bytes, so only first 49 bytes are semantically checked.

Why this step:
- Converted static reverse insight into concrete recovered credential candidate.

Step 7: Validate compared length and exact accepted prefix
Commands run:
- Python len(full_string)
- Python print(full_string[:49]) and len(full_string[:49])

Raw output:
- Full decoded string length: 50
- First 49 bytes:
  flag_{Lin4_D3bugg1ng_iS_MucH_f@5t3r_For_th1$_On3!
- Prefix length: 49

Interpretation:
- Strictly validated bytes are 49-long prefix.
- The final "}" exists in decoded material but is not part of the memcmp length in validator path.

Why this step:
- Prevents off-by-one mistakes and clarifies exact acceptance behavior.

Final Findings
- Recovered candidate (decoded full text):
  flag_{Lin4_D3bugg1ng_iS_MucH_f@5t3r_For_th1$_On3!}
- Exact compared 49-byte password region in validator:
  flag_{Lin4_D3bugg1ng_iS_MucH_f@5t3r_For_th1$_On3!

Cross-Tool Confidence Notes
- Ghidra and Cutter independently corroborated:
  - function addresses,
  - argument flow into validator,
  - RC4-like transform structure,
  - fixed compare length (0x31).
- Cutter decompiler had overlap warnings, but assembly-level behavior matched Ghidra's interpretation.

Assumptions and Limits
- Analysis was static and did not execute malware payload.
- Result is based on reverse-engineered validation path observed in loaded sample.
- If alternate code paths exist (e.g., network C2 override or anti-analysis branch), they were not needed for this specific password gate recovery.
