1) SCOPE AND TOOLS
--------------------------------------------------------------------------------
- Primary tools: Ghidra MCP (strings, xrefs, decompile) and Cutter MCP
  (decompile, string xrefs, segment/header context from prior logs).
- Goal: Find authentication string referenced by malware sample; user
  hypothesis: password to shut down or unlock behavior exists in-file.
- Approach: String-guided reverse engineering -> validation routine ->
  cryptographic transform -> invert transform offline.


2) STEP 1 — BROAD STRING AND ENTRY SEARCH (PARALLEL QUERIES)
--------------------------------------------------------------------------------
Actions taken (batch, for speed):
- Ghidra: list_strings filter "pass"  -> one strong hit.
- Ghidra: list_strings filter "shut"  -> libc / error noise only.
- Ghidra: list_strings filter "kill"  -> RF-kill error string only.
- Ghidra: search_functions_by_name "main" -> no symbol named main.
- Cutter: search_functions_by_name "main" -> no match (stripped binary).

Data read (Ghidra, filter "pass"):
  Address    String
  00478068   "Usage: %s <password>\n"

Interpretation:
- A printf-style usage string almost always marks the real CLI gate for a
  secret argument. Competing "shutdown" / "kill" substrings did not surface
  application-specific passwords; the usage line became the anchor.

Why this mattered:
- Stripped static binaries hide "main"; strings + xrefs are faster than
  walking the entire call graph from the ELF entry.

Action: Ghidra decompile_function_by_address(0x004026f4)

Data read (decompiler output, summarized):
  int FUN_004026f4(int param_1, undefined8 *param_2)
  {
    int local_c;
    if (param_1 == 2) {
      local_c = FUN_004024b8(param_2[1]);
      if (local_c == 1)
        FUN_0040bd80("You achieved Level 8!");
      else
        FUN_0040bd80("Sorry, not quite there yet.");
    } else
      FUN_00405c20("Usage: %s <password>\n", *param_2);

  }

Interpretation:
- param_1 == 2 means argc == 2: program name + one password argument.
- param_2[1] is argv[1], passed to FUN_004024b8 for verification.
- Return value 1 => success message; else failure message.
- This is a CTF-style "Level 8" unlock, not necessarily OS-level ransomware exit,
  but it is the in-binary secret the user asked for.

Why this mattered:
- Pinpointed the single callee that contains the real check: FUN_004024b8.


4) STEP 3 — DECOMPILE THE VALIDATION ROUTINE (FUN_004024b8)
--------------------------------------------------------------------------------
Action: Ghidra decompile_function_by_address(0x004024b8)

Data read (logical structure; full decompiler listed many stack byte inits):
- Registers signal handlers early: FUN_004050c0(5, FUN_00401785); FUN_00405180(5);
- Time-related paths: FUN_00426060 for timestamps; computes duration local_20;
  if local_20 > 500 (interpreted as ms threshold in decompilation), returns false.
- Anti-analysis calls before the crypto path: FUN_00402464(), FUN_00401c2e();
  both must succeed (return 0) to proceed (decompiler shows compound condition).
- Copies argv-derived input into stack slots local_c8 .. (53 bytes of user data).
- Calls FUN_00401a89(&local_c8, 0x35, 0x9988776655443322):
    * Buffer address, length 0x35 (53 decimal), third argument a 64-bit constant key.
- Compares transformed buffer to a 53-byte constant array on stack (local_88
  down through local_54 in Ghidra naming): uses thunk to resolved memcmp.
- Returns true iff compare is equal (iVar1 == 0 after memcmp-style call).

Interpretation:
- Password is exactly 53 bytes from the program's perspective (or the copy is
  fixed-width and compared as 53 bytes; behavior for shorter inputs depends on
  surrounding copy logic—here the compare size is explicitly 0x35).
- Transform is keyed: constant 0x9988776655443322 is not the plaintext password;
  it seeds the stream cipher / XOR layer.
- Reference ciphertext is embedded as immediate byte values in the function,
  not as a readable rodata string—typical for obfuscated flags.

Why this mattered:
- Reduced the problem to: recover P such that
    Transform(P, key=0x9988776655443322) == C
  where C is the 53-byte stack constant array.
- If Transform is XOR with a keystream K derived only from key and counter,
  then P = C XOR K (inversion without brute force).


5) STEP 4 — IDENTIFY THE TRANSFORM (FUN_00401a89)
--------------------------------------------------------------------------------
Action: Ghidra decompile_function_by_address(0x00401a89)

Data read (high level):
- Builds an 8-byte key schedule from param_3 by shifting bytes:
  for i in 0..7: byte_i = (param_3 >> ((7 - i) * 8)) & 0xff
  So the key in memory order (low address first) is MSB-to-LSB of the 64-bit
  value: for 0x9988776655443322 -> bytes 99 88 77 66 55 44 33 22.
- Places those bytes into a 64-bit stack variable; mixes with rodata constant
  string pattern associated with "expand 8-byte k" (Salsa20 sigma variant).
- Zero-initializes two 16-byte areas; loop variable local_18 acts as block
  counter (0, 1, 2, ... per 64-byte output block).
- Each iteration: FUN_00401799(local_a8, state) produces 64 bytes keystream
  into local_a8; XORs keystream into param_1[offset ..] for length up to 64
  or remaining length.
- Anti-tamper: if DAT_004a50d0 != 0, instead of XOR, calls a routine that
  zeroes the buffer (thunk_FUN_00419390 style path)—would defeat verification
  under instrumentation that sets that flag.

Interpretation:
- Classic stream cipher usage: ciphertext_stream = plaintext_stream XOR KS(key,nonce,counter).
- Here the "encryption" applied to the user input before compare means:
    C = P XOR KS
  so P = C XOR KS for the same KS.

Action: Cutter decompile_function_by_address(0x00401a89) for clearer struct layout

Data read (Rizin/rz-ghidra naming, critical constants):
- var_20h points to data at 0x00478010 (sigma / k string region).
- After key byte loop, explicit CONCAT/assignments expose 32-bit words:
    * 0x61707865  ("expa" LE)
    * 0x3820646e  (fragment of "expand 8-byte k")
    * 0x7479622d  ("-byt" LE)
    * 0x006b2065  ("e k" with leading zero byte in the word)
- Key words: low 32 bits k1 = 0x22334455, high 32 bits k0 = 0x66778899 when
  the 8-byte key MSB-first extraction is stored little-endian as uint64
  0x2233445566778899 on the stack.

Why this mattered:
- Cutter's CONCAT44 layout disambiguated the order of the sixteen 32-bit
  Salsa20 state words better than Ghidra's overlapping stack simplication.


6) STEP 5 — CONFIRM CORE PRIMITIVE (FUN_00401799)
--------------------------------------------------------------------------------
Action: Ghidra decompile_function_by_address(0x00401799)

Data read:
- Copies 16 words from input state via thunk_FUN_00419090 (Ghidra names resolve
  oddly in some builds; logic is still visible).
- Applies 10 "double rounds" of quarter-round updates with rotations 7, 9, 13,
  18 bits—matching the Salsa20 / ChaCha family column-round then row-round
  structure.
- Final step: output_word[i] = original_state[i] + working_state[i] (mod 2^32),
  packed into 64 output bytes.

Interpretation:
- This is the Salsa20 block function (20 rounds = 10 double rounds).
- Not a custom ad-hoc hash; standard structure means a vanilla implementation
  can reproduce keystream if state words match the binary's layout.

8) STEP 7 — RECONSTRUCT SALSA20 STATE WORDS (TRIAL AND ERROR TO MATCH BINARY)
--------------------------------------------------------------------------------
Initial attempt (incorrect word order for words 0-7):
- Placed constants and k0,k1 in an order that swapped some sigma/key slots.
- Python XOR produced garbage (non-printable first byte 0x8f).

Correction:
- Align 16 x uint32 little-endian state exactly as memory layout from Cutter:
  w0  = 0x61707865  (sigma)
  w1  = k1          (0x22334455)
  w2  = k0          (0x66778899)
  w3  = 0x3820646e
  w4  = 0x7479622d
  w5  = k1
  w6  = k0
  w7  = 0x006b2065
  w8  = block_counter & 0xffffffff  (counter = 0 for first block)
  w9  = (block_counter >> 32) & 0xffffffff
  w10..w15 = 0

Interpretation:
- Salsa20 output depends critically on word order; one transposed pair breaks
  the entire keystream.
- Block counter  suffices for 53 bytes (only first keystream block needed).
