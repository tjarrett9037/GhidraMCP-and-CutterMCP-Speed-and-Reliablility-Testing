Ransomware Shutdown Password Recovery Log
Date: 2026-03-31
Analyst: Cursor agent session

Objective
- Identify the password embedded in a ransomware sample loaded in Ghidra that can be used to shut down / stop malware behavior.

Method Summary
- Use Ghidra MCP tools to enumerate strings and symbols.
- Locate likely password-validation code path.
- Decompile relevant functions.
- Reconstruct transform/check routine.
- Recover plaintext password by reproducing algorithm externally.

============================================================

============================================================

Step 1: Initial hypothesis and search strategy
What I did:
- Started with string triage for words commonly associated with shutdown and credential checks.
- Queried strings for the following filters: "pass", "shutdown", "stop", "kill".
- Queried imports and exports to identify entrypoints and any obvious external crypto/API helpers.

Why:
- In many malware/CTF binaries, password prompts and usage strings leak the validation path.
- "shutdown/kill/stop" often surfaces kill-switch messages or admin command handlers.

Step 4: Decompile and interpret argument-check wrapper

4.1 Query: decompile_function_by_address("0x00401b46")
Returned decompiled code:

int FUN_00401b46(int param_1,undefined8 *param_2)
{
  int local_c;
  
  if (param_1 == 2) {
    local_c = FUN_00401a09(param_2[1]);
    if (local_c == 1) {
      FUN_0040a590("You achieved Level 5!");
    }
    else {
      FUN_0040a590("Sorry, not quite there yet.");
    }
  }
  else {
    FUN_00404c20("Usage: %s <password>\n",*param_2);
  }
  return local_c;
}

Interpretation:
- Expects exactly one argument after program name (argc == 2).
- Calls FUN_00401a09 on argv[1], which is the user-supplied password.
- Success path prints "You achieved Level 5!", failure prints "Sorry, not quite there yet."
- Therefore FUN_00401a09 is the core password validator.
- Next action: decompile FUN_00401a09.

------------------------------------------------------------
Step 5: Decompile and interpret password validator

5.1 Query: decompile_function_by_address("0x00401a09")
Returned key behavioral data:
- A 47-byte constant is loaded into local bytes local_48 ... local_1a:
  5e de b9 d4 d8 8a b2 ca 97 20 98 a3 cb b2 ab d6
  56 b9 b7 95 c4 33 91 1b 54 60 9e 69 b3 75 5c da
  05 16 00 5a 67 b3 3f 41 6c 3f 3c 64 a4 3d cd
- Input bytes are copied into a local buffer (beginning at local_78), total length used: 0x2f (47 bytes).
- Calls:
  FUN_00401725(&local_78, 0x2f, 0x5f5d6e4c3b2a1d0e)
- Then compares transformed input against constant bytes:
  thunk_FUN_00416c20(&local_78, &local_48, 0x2f)
- Returns true iff comparison == 0.

Interpretation:
- Validation logic is:
  transform(user_input_47_bytes, key=0x5f5d6e4c3b2a1d0e) == embedded_47_byte_constant
- Need to reverse/understand FUN_00401725 transform routine.

------------------------------------------------------------
Step 6: Decompile and classify the transform algorithm

6.1 Query: decompile_function_by_address("0x00401725")
Key algorithm observations from decompilation:
- Initializes S-box array 0..255.
- Performs key scheduling over 256 rounds using 8-byte key material.
- Generates pseudo-random byte stream over requested length.
- XORs generated keystream with input buffer in-place.

Interpretation:
- This is RC4-like KSA + PRGA behavior.
- Because operation is XOR stream cipher, decrypt and encrypt are the same:
  plaintext XOR keystream = ciphertext
  ciphertext XOR keystream = plaintext
- Therefore we can recover expected plaintext password by applying same keystream to embedded constant.

------------------------------------------------------------
Step 7: Reproduce algorithm and recover plaintext password

7.1 External verification script run in Python
Action:
- Implemented RC4-like routine mirroring decompiled logic.
- Parsed key from constant 0x5f5d6e4c3b2a1d0e as little-endian bytes.
- Used embedded 47-byte constant as ciphertext.
- XORed with generated keystream to recover plaintext.

Script output:

Interpretation:
