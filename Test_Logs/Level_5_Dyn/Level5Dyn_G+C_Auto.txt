================================================================
3) FUNCTION-LEVEL ANALYSIS
================================================================
3.1 Ghidra decompile: FUN_0010157a (main-like)
Critical behavior:
- if argc == 2: call FUN_0010143d(argv[1])
- if true -> puts("You achieved Level 5!")
- else -> puts("Sorry, not quite there yet.")
- else (argc != 2) -> printf("Usage: %s <password>\n", argv[0])

Interpretation:
- This is the command entry gate.
- Password checking happens in FUN_0010143d.

3.2 Ghidra decompile: FUN_0010143d
Observed:
- Loads a fixed 47-byte constant byte array into stack locals.
- Copies user input buffer into local stack region.
- Calls FUN_00101159(&local_input, 0x2f, 0x5f5d6e4c3b2a1d0e).
- Executes: memcmp(&local_input, &constant_bytes, 0x2f).
- Returns true when memcmp == 0.

Critical extracted values:
- Compare length: 0x2f (47 bytes).
- Transform key immediate: 0x5f5d6e4c3b2a1d0e.
- Target bytes (47 total):
  5e de b9 d4 d8 8a b2 ca
  97 20 98 a3 cb b2 ab d6
  56 b9 b7 95 c4 33 91 1b
  54 60 9e 69 b3 75 5c da
  05 16 00 5a 67 b3 3f 41
  6c 3f 3c 64 a4 3d cd

Interpretation:
- The binary does not compare plaintext directly.
- It transforms input then compares against embedded ciphertext-like bytes.

3.3 Ghidra + Cutter decompile: FUN_00101159 / fcn.00001159
Algorithm characteristics observed:
- Initializes S-box with values 0..255.
- Key scheduling loop uses 8-byte repeating key.
- PRGA loop generates keystream bytes.
- XORs keystream with input buffer in-place.

Interpretation:
- This is RC4-style stream cipher logic.
- Therefore:
  transformed_input = RC4(input, key)
  compare transformed_input to fixed bytes
- To recover expected password:
  password = RC4_decrypt(fixed_bytes, key)
  (For RC4, encrypt/decrypt is same XOR stream operation.)

3.4 Cutter cross-check (fcn.0000143d and fcn.00001159)
Findings:
- Same constant byte array present.
- Same transform invocation with key immediate 0x5f5d6e4c3b2a1d0e.
- Same memcmp length 0x2f.

Interpretation:
