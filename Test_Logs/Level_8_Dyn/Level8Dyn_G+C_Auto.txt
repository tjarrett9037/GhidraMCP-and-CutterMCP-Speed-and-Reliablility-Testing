STEP 4 — DECOMPILE THE PASSWORD CHECK (FUN_00101f7c)
================================================================================

Action
------
• Ghidra: decompile_function_by_address 0x00101f7c

Data read — key behavioral elements (summarized from decompilation)
--------------------------------------------------------------------
• signal(5, FUN_00101249); raise(5);
• clock_gettime(1, &start_time)
• A 53-byte (0x35) byte array is filled with constants on the stack (target for memcmp).
• User input is copied into a 53-byte stack buffer (structured as qword chunks in the decompiler).
• FUN_0010154d(&user_buffer, 0x35, 0x9988776655443322) mutates the user buffer in place.
• clock_gettime again; elapsed time must be ≤ 500 ms or check fails.
• memcmp(transformed_user, expected_53_bytes, 0x35) must be zero for success.
• Prior gates: FUN_00101f28 and FUN_001016f2 must both return 0 (anti-debug / environment).

Numeric / structural data explicitly important
-----------------------------------------------
• Transform call: FUN_0010154d(..., length=0x35, third argument=0x9988776655443322).
• Compare length: 0x35 = 53 bytes.

Interpretation
--------------
• The password is NOT compared in plaintext; it is passed through FUN_0010154d
  (stream-like XOR after a keystream generator) and compared to an embedded
  53-byte constant.
• Runtime constraints (signal, ptrace-style checks, 500 ms) explain why running
  under a debugger or slowly might fail even with the correct password.
• The 64-bit constant 0x9988776655443322 is a fixed “seed” or key material
  for keystream generation (endian-extracted into bytes in the next function).

Why
---
memcmp after an in-place transform tells us the recovery problem: invert or
reimplement the transform to XOR ciphertext (expected blob) with the same
keystream to recover the passphrase.

STEP 5 — DECOMPILE FUN_0010154d (STREAM APPLICATION)
================================================================================

Action
------
• Ghidra: decompile_function_by_address 0x0010154d

Data read — algorithm structure
-------------------------------
• Zeros a 64-byte stack region (conceptually), then writes:
  - Bytes extracted from third parameter (8 bytes, one per loop iteration using shifts).
  - Interleaved with dwords from the string at 0x00103004 ("expand 8-byte k" / 16 bytes as 4 dwords).
• Outer loop over data length in steps of 0x40 (64), with a block counter written
  into the key state (later verified via disassembly at RBP-0x40 / -0x3c).
• Inner: FUN_0010125d(keystream_out, key_state); then for each byte in the current
  chunk: buffer[i] ^= keystream[i].
• If a global (DAT_001050a0) is non-zero, the function memsets the user buffer to
  zero (anti-debug / tamper path).

Interpretation
--------------
• This is “generate 64 bytes of keystream from a 64-byte expanded key state,
  XOR into plaintext/ciphertext buffer, advance block counter.”
• Because XOR is symmetric, if C = P XOR KS, then P = C XOR KS when KS matches.
• The expected memcmp blob is the post-transform value; XORing it with the
  same keystream bytes yields the original password bytes.

Why
---
Establishes exact operations needed for offline recovery without running the
possibly hostile binary.

STEP 6 — DISASSEMBLE FUN_0010154d FOR EXACT MEMORY LAYOUT (GHIDRA MCP)
================================================================================

Action
------
• Ghidra: disassemble_function address 0x0010154d

Data read — machine-level facts used later
------------------------------------------
• PXOR + MOVAPS clear 64 bytes at RBP-0x60 .. (four xmmword stores).
• LEA to 0x00103004 loads the “expand 8-byte k” string address.
• SAR on RDX with (7-c)*8 extracts bytes of the third argument into a temporary,
  stored at RBP-0x20 downward — i.e. MSB-to-LSB order into memory for the first
  qword layout used as two little-endian dwords.
• Sequenced MOVs place dwords into the 64-byte key block at RBP-0x60:
  pattern: magic dword, key dword0, key dword1, magic dword, magic dword,
           key dword0, key dword1, magic dword (8 dwords = 32 bytes).
• Each outer iteration:
  MOV qword at RBP-0x10 (block index) into dwords at RBP-0x40 and RBP-0x3c.
• CALL 0x0010125d with RDI = keystream buffer (RBP-0xa0), RSI = key block (RBP-0x60).

Interpretation
--------------
• First 32 bytes of the 64-byte key state are fixed from string + key constant;
  next 8 bytes are the 64-bit block counter (low 32 at -0x40, high 32 at -0x3c);
  remaining bytes stay zero from the initial clear — total 64 bytes fed to the mixer.
• For a 53-byte password only block index 0 is needed (53 < 64).

Why
---
Decompilers sometimes blur stack layout; disassembly confirmed counter placement
and zero-filled tail, preventing wrong keystream from off-by-something layout bugs.

STEP 7 — FUN_0010125d: GHIDRA DECOMPILE VS. DISASSEMBLY
================================================================================

Action
------
• Ghidra decompile_function_by_address 0x0010125d showed multiply/shift mixing.
• Ghidra disassemble_function 0x00100125d / 0x0010125d revealed ROL/ROR/XOR ops.

Data read — ground truth from disassembly (conceptual)
--------------------------------------------------------
• memcpy local 64 bytes from input key state to working state at RBP-0x50.
• Loop variable at [RBP-0x4]: compare to 9, JLE → 10 total rounds (0..9).
• Each round is a long sequence of 32-bit operations using ROL imm8 and ROR imm8.
• After rounds, for i = 0..15:
  output_dword[i] = input_dword[i] + mixed_dword[i] (ADD, store to first arg buffer).

Interpretation
--------------
• Ghidra’s high-level pseudocode for this function was misleading (wrong model).
• Cutter/Ghidra disassembly is authoritative: custom ARX (add-rotate-xor) mixer.
• Final keystream words are (original_key_state_word + mixed_state_word) mod 2^32,
  serialized little-endian to 64 bytes.

Why
---
Trusting the wrong pseudocode led to garbage plaintext; switching to disassembly
was necessary for correctness.

STEP 8 — TRANSCRIBING ONE FULL MIX ROUND (16×uint32 STATE)
================================================================================

Method
------
• Mapped stack slots RBP-0x50 .. RBP-0x14 to indices v[0]..v[15] (each slot 4 bytes).
• Transcribed each instruction group in order as it appears in the disassembly dump,
  applying updates sequentially (later instructions see updated registers / stack).

Pitfall encountered and fix
----------------------------
• The fourth quaternion in the round (starting ~0x00101375) mixes v[3], v[7], v[11],
  v[15] in an order that differs from the “(0,4,8,12) style” blocks.
• An initial transcription mistakenly used v[15]^=f(v[3],v[7]) style ordering;
  aligning with the actual operands (v[3]^=rol(v[11]+v[15],7), etc.) made the
  recovered plaintext ASCII.

Why order matters
-----------------
Each step reads the current memory; wrong order produces a different permutation
and wrong keystream.

STEP 9 — OFFLINE RECOVERY SCRIPT (PYTHON)
================================================================================

Inputs encoded from analysis
----------------------------
• Third-parameter key: 0x9988776655443322 → eight bytes via SAR-style extraction
  (high to low byte order in the temporary) → little-endian dword pair (k0, k1).
• Magic: sixteen bytes from "expand 8-byte k\0" interpreted as four little-endian dwords.
• 64-byte key state construction:
  words 0..7 from disassembly layout; qword at byte offset 32 = block index (0);
  bytes 40..63 = zero.
• Keystream: for block 0, compute mixed = 10 * mix_one_round on v initialized from
  the 16 input dwords; output dword i = (inp[i] + mixed[i]) & 0xffffffff; flatten LE.
• Plaintext byte i = ciphertext_byte i XOR keystream_byte i (for i in 0..52).

Ciphertext constant (53 bytes) taken from Ghidra stack init in FUN_00101f7c
(hex, in stack order local_88 descending through local_54 as decompiler showed):

  c2 d4 e1 49 bd 6f b4 61 93 3f 77 26 b1 ff 4f e1
  9f 50 03 43 8a c8 65 95 71 1f 9e a5 ab 4e a4 23
  45 08 61 11 64 c8 ff 41 66 bb fa 40 2c 56 b8 ce
  61 be 96 0d 35

Note: decompiler displayed some bytes as decimal (e.g. 200 = 0xC8, 100 = 0x64);
those were normalized to hex when building the byte array.

Result
------
Recoverable ASCII (the password / flag):

  flag_{Lin8_C@n_y0u_M4k3_a_b1n@ry_Pr!nt_!t$_0wn_Fl4g?}

Interpretation
--------------
• This is the preimage under FUN_0010154d of the embedded 53-byte constant.
• It is what argv[1] must equal for memcmp success, modulo anti-debug/timing.
