5.1 Ghidra — STRUCTURE AND LOGIC (FUN_004023a8)
--------------------------------------------------------------------------------
Observed behavior (paraphrased from decompiler output):

A) Setup / environment checks:
   - FUN_00404fa0(5, FUN_00401785);
   - FUN_00405060(5);
   - FUN_004250e0(1, &local_28);   // timestamp / time sample A

B) Embedded 52-byte target buffer (local_78), initialized byte-by-byte in the function body.
   First bytes shown in decompilation:
   local_78[0]  = 0x41 ('A')
   local_78[1]  = 0x9c
   ... through index 0x33 (51 decimal) — total length 0x34 (52) bytes.

C) Guard conditions:
   - iVar1 = FUN_00402354(); must be 0
   - iVar1 = FUN_00401b1e(); must be 0
   If either fails, return 0.

D) Timing gate:
   - FUN_004250e0(1, &local_38);   // time sample B
   - local_18 = (local_38 - local_28) * 1000 + (local_30 - local_20) / 1000000;
   - If local_18 > 500, return 0.
   INTERPRETATION: Validation must complete within ~500 ms — anti-debug / anti-slow-analysis.

E) Copy user input:
   - For local_c from 0 to 0x33 (52 iterations):
       local_b8[local_c] = *(char *)(param_1 + local_c);
   INTERPRETATION: Reads up to 52 bytes from the password string pointer (not necessarily
   null-terminated in the loop; comparison length is fixed at 52).

F) Transform in place:
   - FUN_00401974(local_b8, 0x34, 0x123456789abcdef);
   INTERPRETATION: 52-byte buffer, third argument is a 64-bit key constant.

G) Constant-time-style byte compare:
   - Loop local_10 from 0; while local_10 <= 0x33:
       if local_b8[local_10] != local_78[local_10]: break and return 0
   - If all match, return 1.

CONCLUSION FROM 5.1:
- The password is not stored as plaintext in the binary.
- The program stores an obfuscated/encrypted 52-byte target (local_78).
- User input is transformed with FUN_00401974 using key 0x0123456789ABCDEF (same numeric value
  as written in decompilation; endianness of the constant is handled inside the transform).

6. TRANSFORM FUN_00401974 AND DEPENDENCIES
================================================================================
6.1 Ghidra — FUN_00401974(long param_1, int param_2, long param_3)
--------------------------------------------------------------------------------
Observed logic (summary):

- If DAT_004a50d0 != 0: alternate path (thunk to memset-like zeroing) — not taken in normal
  static analysis path if that flag is zero.

- Build 8 bytes from param_3 (64-bit key), extracting bytes with shifts:
    auStack_20[i] = (char)(param_3 >> (((7 - i) * 8) & 0x3f))

- Pack into four 16-bit values (local_28[0..3]) as big-endian pairs from those bytes.

- Counter local_18 starts at 0; offset local_c starts at 0.
- While local_c < param_2 (52):
    - Fill local_30[0..3] from local_18 (counter split into four 16-bit lanes).
    - FUN_004017fa(local_28, local_30) — mix counter block with key schedule material.
    - For local_10 in 0..7 (and within buffer length):
        XOR each byte at param_1 + local_c + local_10 with a byte derived from local_30:
        - Even local_10: use high byte of local_30[local_10/2]
        - Odd local_10: use low byte of local_30[local_10/2]
    - local_18++; local_c += 8

INTERPRETATION:
- Stream-like XOR: transformed[i] = input[i] XOR keystream_byte(i).
- Keystream depends on key param_3 and block index (local_18).

6.2 Ghidra — FUN_004017fa(undefined2 *param_1, ushort *param_2)
--------------------------------------------------------------------------------
- param_1: pointer to four key ushorts (from the 8-byte key expansion).
- param_2: four ushort state words; updated in place over 8 rounds.
- Each round uses FUN_00401799 repeatedly (ARX-like mixing with multiply primitive).

INTERPRETATION:
- Custom lightweight block function generating pseudo-random-looking 8-byte keystream per block
  from counter + key.

6.3 Ghidra — FUN_00401799(ushort param_1, ushort param_2)
--------------------------------------------------------------------------------
- Implements a 16-bit multiply with special correction:
  - uVar1 = (uint)param_1 * (uint)param_2 (with zero guards).
  - Combines low and high 16 bits of the 32-bit product with an extra decrement when product
    exceeds 16 bits.

INTERPRETATION:
- Non-standard multiplication mod 2^16 with overflow handling — must be bit-accurate in any
  reimplementation.

7. SOLVING FOR THE PLAINTEXT PASSWORD
================================================================================
7.1 Algebraic reasoning
--------------------------------------------------------------------------------
Let P = 52-byte password (as read into the buffer).
Let T = 52-byte embedded target (local_78).
FUN_00401974 XORs P with keystream KS(key, block_index) in 8-byte blocks.

Therefore:  P[i] = T[i] XOR KS[i]  for i in 0..51,
where KS is determined solely by key 0x0123456789ABCDEF and the block counter.

7.2 Implementation approach (session work)
--------------------------------------------------------------------------------
A Python script was written during the session to:
- Implement FUN_00401799, FUN_004017fa, and the keystream extraction from FUN_00401974 with
  C operator precedence matched for the round function (notably + versus ^).
- Generate 8-byte keystream blocks for counters 0, 1, ... covering 52 bytes.
- XOR the known target T to recover P.
